IBR — Informações de Seletores (Linhas + Variáveis)
Versão: 2.3 | Data: 2026-02-19

Uso permitido
- Este arquivo serve apenas para roteamento interno (decidir qual consulta oficial/catálogo usar).
- É proibido usar como fonte para concluir linha/tamanho/limites sem validação oficial.

FIREWALL (OBRIGATÓRIO) — nunca exibir ao cliente
Proibido expor: nomes de seletores, chaves internas, tokens, variáveis, ids, enum internos.
Uso permitido: somente para roteamento interno (decidir qual catálogo/configurador consultar).

TRADUÇÃO PARA LINGUAGEM DO CLIENTE
Sempre converter termos internos para rótulos humanos.
Nunca imprimir o termo interno original.

OBJETIVO
- Centralizar a taxonomia de seletores, linhas internas e variáveis do configurador.
- Padronizar escrita, apelidos, sinônimos e observações para BUSCA, auditoria e validação.
- Servir como referência de leitura. A verdade final de código, referência, compatibilidade e link é sempre Action/API.

REGRAS DE SEGURANÇA (obrigatórias)
- Este arquivo NÃO autoriza montar referência/código manualmente fora do seletor correspondente.
- Nunca inferir valores ausentes. Qualquer campo dependente de consulta auxiliar deve vir do retorno oficial.
- Em ambiguidade, NÃO decidir por texto. Validar por Action/API.
- Nunca usar "descricao" para decidir token. Decisão sempre por id/variavel/sinônimos normalizados + validação.
- Nunca promover sinônimos a token oficial. Sinônimos existem somente para busca.
- USO INTERNO: este arquivo é para roteamento/triagem/auditoria. Nunca expor ao usuário final nomes de seletores, tokens ou variáveis internas.
- Saída para o cliente deve ser sempre em linguagem amigável (ex.: “flange IEC B14”, “montagem”, “redução”), sem códigos internos.

NORMALIZAÇÃO (para busca)
- Converter para UPPERCASE.
- Remover acentos somente para busca (não regravar como dado oficial).
- Colapsar espaços internos e remover espaços ao redor de separadores.
- Preservar '.', '/', '+', '-', '_' quando fizerem parte de código/referência.
- Preferir match por boundary quando possível (evita PC3 bater em PC30).

ESTRUTURA DO ARQUIVO
- [META] informações de versão, origem e regras.
- [INDICE] contadores e lista de seletores.
- [SELETOR] cadastro dos seletores (nível 1 do configurador).
- [LINHA] linhas internas (sub-linhas) por seletor, quando existirem.
- [VAR] cadastro de variáveis por seletor (produto_pai), com sinônimos e controles.

GRAMÁTICA E CAMPOS (contrato)
- Campos OFICIAIS: codigo, id, produto_pai, variavel. Estes são os únicos campos que podem virar decisão/consulta.
- Campos AUXILIARES: apelidos, sinonimos, observacoes, regras_chave, risco_colisao, ui_label.
- Datas: YYYY-MM-DD.
- Status (enum): ativo | inativo | legado | rascunho.
- versao_item: versão do item.
- depends_on: lista de dependências (tokens/consultas auxiliares).
- requires_api: true/false. Se true, a resposta final depende de Action/API.

REGRAS DE DESAMBIGUAÇÃO (ordem de prioridade)
1) Match exato em id (ex.: FX.FXOPX).
2) Match exato em variavel dentro do produto_pai (ex.: FXOPX).
3) Match exato em sinonimos_norm (termo normalizado).
4) Match parcial com boundary + maior especificidade (peso maior).
5) Se houver empate ou risco, parar e validar por Action/API.

REGRAS DE BLOQUEIO (anti-colisão)
- Termos curtos (1–2 letras) não decidem sozinhos: C, Q, R, P, IN.
- Termos com possível colisão de linguagem comum devem exigir contexto:
  - PES: pode colidir com "peso". Só aceitar se contexto indicar montagem/apoio.
  - CAPA/TAMPA/COVER: exigir contexto de proteção/tampa.
  - POSICAO/ORIENTACAO: exigir referência ao item correto (fixação/saída/montagem).
- Se risco_colisao = alto, sempre requires_api = true (ou bloquear decisão por texto).

INTEGRIDADE (checks recomendados)
- id é único globalmente.
- produto_pai existe em [SELETOR].
- variavel não se repete dentro do mesmo produto_pai, exceto sufixos explícitos (ex.: AEEE/AEEE2).
- Se ui_label existir, deve mapear para uma variável exibida em UI, mas a decisão continua pelo id/variavel oficial.
- Linhas internas, se usadas para decisão, devem declarar aplica_em (seletor(es) válidos).

[REGRAS_ESPECIAIS_DE_ROTEAMENTO_TEXTUAL]
objetivo: resolver colisões entre descrição comercial e aplica_em compartilhado sem inferir seletor final por texto cru

regras_gerais:
- aplica_em compartilhado não decide sozinho o seletor final
- descrição comercial serve para pré-roteamento, nunca para conclusão final sem Action/API
- quando existir regra explícita de roteamento textual, ela prevalece sobre aplica_em compartilhado
- ordem interna de linhas de um seletor não pode ser reutilizada como heurística global

regras_explicitas_confirmadas:
- "IBR P FFA" -> pré-rotear como FX
- "IBR P" sem complemento conflitante -> pré-rotear como FX
- aplicar o mesmo princípio para regras explícitas próprias de "IBR C" e "IBR H", sem derivar uma pela outra

bloqueios:
- é proibido classificar item como HY apenas porque a descrição contém FFA, FKA, FR ou R
- HYLN só pode ser avaliado depois que o item já estiver oficialmente confirmado como HY
- em caso de conflito entre descrição comercial e enquadramento, bloquear conclusão e validar por Action/API

observacao:
- este bloco organiza pré-roteamento textual
- a conclusão final de seletor, linha e elegibilidade continua dependente de retorno oficial
