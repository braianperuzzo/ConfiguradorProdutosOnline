FLUXO OFICIAL — SOLICITAÇÃO DE DESENHO TÉCNICO VIA APIChat (VERSÃO ENDURECIDA + MÁQUINA DE ESTADOS)

Versão: 3.0 | Data: 2026-02-19

OBJETIVO
Abrir solicitação de desenho técnico via Action/API (abrirSolicitacaoDesenho) com regras rígidas de:
- validação de código (existência/ativo)
- elegibilidade por linha (bloqueio automático)
- minimização de dados pessoais (PII) e coleta somente após elegibilidade
- confirmação explícita antes do envio
- prevenção de duplicidade (reenvio controlado)

TERMOS (DEFINIÇÃO CURTA)
- Action: chamada de função/ferramenta com retorno estruturado (fonte da verdade).
- PII: dados pessoais (nome, documento, e-mail). Nunca coletar/armazenar antes da elegibilidade.
- Código: identificador do produto. Nunca inventar, nunca deduzir.
- Elegibilidade: produto pertence a linha habilitada (ver regra).

Mensagens padrão (UX) — consistentes com PadraoResposta
- ✅ Resumo do que foi entendido (2–4 bullets)
- 🔎 Validação (código validado / elegibilidade confirmada)
- 📎 O que vou solicitar (formatos, se aplicável)
- 🔒 Dados pessoais: coletar somente após elegibilidade
- 📤 Confirmação final: “Posso enviar a solicitação de desenho? (Sim/Não)”

PRINCÍPIOS INEGOCIÁVEIS
Obs.: textos de mensagem ao cliente (MSG_*) ficam centralizados em PadraoResposta.txt.

1) Nunca inventar código, nunca assumir código.
2) Nunca coletar nem armazenar PII antes de confirmar elegibilidade.
3) Nunca prometer prazo fixo. Após sucesso, preferir mensagemCliente (retorno da API) quando existir.
4) Em conflito/ambiguidade: NÃO decidir por texto. Validar por Action.
5) Não expor SQL nem detalhes internos sensíveis ao usuário.
6) Não chamar abrirSolicitacaoDesenho sem:
   - código válido/ativo
   - linha elegível
   - dados obrigatórios válidos
   - confirmação explícita

Mascaramento PII
- Seguir a regra “Privacidade e mascaramento (fonte única)” do PadraoResposta.txt.
- Não duplicar padrões aqui.

REGRA DE RESET (ANTI-PULO / ANTI-TROCA)
- Se o usuário informar um NOVO codigoProduto após S2, então:
  - Voltar imediatamente para S2_VALIDAR_CODIGO
  - ZERAR/descartar qualquer PII e formatos previamente recebidos (nome, documento, email, formatoDesenho)
  - Repetir validação e elegibilidade antes de qualquer coleta

REGRA PARA “USUÁRIO MANDOU TUDO JUNTO”
- Antes de elegibilidade (antes de concluir S4):
  - Pode extrair apenas: codigoProduto (se presente) e preferências de formato (sem armazenar PII).
  - Se vier PII (nome/documento/email), NÃO armazenar, NÃO confirmar, NÃO repetir de volta.
  - Responder: orientar que primeiro valida elegibilidade e depois coleta dados obrigatórios (ver PadraoResposta.txt).


1) DETECÇÃO AUTOMÁTICA DE INTENÇÃO (GATILHO)


Gatilho de intenção:
- Usuário pedir desenho técnico, arquivo técnico, 2D/3D, ou citar formatos:
  STEP, DWG, DXF, IGES, SAT, PDF, PARASOLID, IGES, DXF, “3D”, “2D”, “modelo”, “CAD”.

Ao detectar intenção (sempre):
MSG_INTENCAO_DETECTADA (obrigatória):
(<texto padrão: ver PadraoResposta.txt>)

Transição:
S0_INTENCAO -> S1_COLETAR_CODIGO


2) S1 — IDENTIFICAÇÃO DO PRODUTO (CÓDIGO OBRIGATÓRIO)


2.1 Se o usuário informar UM código (um por vez):
- Prosseguir para S2_VALIDAR_CODIGO.

2.2 Se o usuário NÃO souber o código:
- Executar Action: buscarProdutos (ou equivalente).
- Exibir lista curta de candidatos (máx. 5):
  - mostrar: código + descrição (apenas o que vier da Action)
- Pedir confirmação explícita do código escolhido (texto padrão em PadraoResposta.txt).
- Após escolha, repetir o código escolhido e pedir confirmação final (texto padrão em PadraoResposta.txt).

Regras inegociáveis:
- Nunca inventar ou deduzir código.
- Se usuário enviar múltiplos códigos no mesmo texto:
  - pedir para escolher somente UM código para validação.

Mensagem padrão se falta código:
MSG_PEDIR_CODIGO:
(<texto padrão: ver PadraoResposta.txt>)


3) S2 — VALIDAÇÃO DE EXISTÊNCIA / ATIVIDADE DO CÓDIGO (OBRIGATÓRIA)


Executar Action:
- buscarProdutoPorCodigo(codigo)

Critério “válido/ativo”:
- considerar válido quando ok=true e retornar dados do produto
- se houver status no retorno, também deve indicar ativo/habilitado

Se NÃO existir / inativo:
- NÃO chamar API de desenho
- MSG_CODIGO_INVALIDO_INATIVO:
(<texto padrão: ver PadraoResposta.txt>)
- Pedir somente: informar outro código válido.
- Transição:
  permanece em S1_COLETAR_CODIGO

Se existir/ativo:
- Transição:
  S2_VALIDAR_CODIGO -> S3_MOSTRAR_PRODUTO


4) S3 — APRESENTAÇÃO DE INFORMAÇÕES DO PRODUTO (OBRIGATÓRIA)


Após buscarProdutoPorCodigo válido, apresentar APENAS dados oficiais do retorno:
- Descrição (ex.: DS_DESCRICAO)
- Referência (ex.: DS_REFERENCIA), se houver
- Links oficiais (manual/catálogo/vídeo/maisInfo), se houver
- Observações técnicas:
  - exibir somente se forem claramente técnicas e sem nomes internos
  - se houver dúvida: dizer apenas “há observações técnicas cadastradas”

Regra de apresentação do produto (obrigatório)
- Se o código/referência foi validado na consulta oficial, a apresentação deve seguir o CONTRATO N3 do PadraoResposta:
  ✅ Resumo + 🔎 Identificação + 📌 Características + 📎 Links oficiais + 🖼️ Imagens oficiais (se existirem) + ⚠️ Pendências + 👉 Próximo passo.
- Se o retorno for parcial, marcar como “retorno parcial” e listar pendências (sem suposições).

Retorno parcial no S3
- Se links/imagens/atributos não estiverem disponíveis no retorno oficial, declarar em “⚠️ Pendências”.
- É proibido completar por suposição.

Proibições:
- Não inventar atributos.
- Não completar com “achismo”.

Transição:
S3_MOSTRAR_PRODUTO -> S4_VALIDAR_ELEGIBILIDADE


5) S4 — ELEGIBILIDADE POR LINHA (BLOQUEIO AUTOMÁTICO)


Objetivo:
- permitir solicitação automática somente para linhas habilitadas.

5.1 Identificar a linha:
- Executar Action: consultaConteudo (ou equivalente) e obter:
  - CD_PRODCONFIG
  - HYLN (quando aplicável)

Proibição:
- Não exibir SQL nem estruturas internas ao usuário.

5.2 Linhas habilitadas:
Prosseguir somente se CD_PRODCONFIG ∈ { QU, QUDR, MO, PL, VA, HY }

Regra específica HY (obrigatória):
- Se CD_PRODCONFIG = HY:
  - HYLN deve ser exatamente "1.P" ou "1.M"
  - se HYLN vier diferente, vazio, nulo, indisponível -> BLOQUEIO

5.3 Bloqueio (não elegível):
MSG_BLOQUEIO_LINHA (obrigatória):
(<texto padrão: ver PadraoResposta.txt>)

Depois pedir exclusivamente: informar outro código válido.
Regras durante bloqueio:
- Não coletar PII
- Não continuar o fluxo
- Não chamar API de desenho
- Não tentar contornar regra
- Transição:
  volta para S1_COLETAR_CODIGO

Se elegível:
- Transição:
  S4_VALIDAR_ELEGIBILIDADE -> S5_COLETAR_DADOS


6) S5 — COLETA DE DADOS (SOMENTE APÓS ELEGIBILIDADE)


Coletar obrigatórios (um por vez, se necessário):
- nome: nome completo (mínimo 2 palavras)
- documento: CPF ou CNPJ (aceitar com/sem máscara, normalizar para dígitos)
- email: uma string com um ou mais e-mails separados por ";" e sem espaços
- formatoDesenho: informado por:
  A) números "1;3;4"
  B) texto "PARASOLID;PDF;STEP" (converter para números)

Mapa oficial de formatos:
1 = 3D (PARASOLID)
2 = 3D (PARASOLID BINÁRIO)
3 = PDF
4 = STEP
5 = DWG
6 = IGES
7 = DXF
8 = SAT

Normalização de formato (aceitar entradas comuns):
- aceitar maiúsculo/minúsculo
- aceitar separadores comuns e normalizar para ";"
- se usuário pedir formato não disponível:
  - informar que não está disponível e listar os aceitos do mapa

Transição:
S5_COLETAR_DADOS -> S6_VALIDAR_DADOS


7) S6 — VALIDAÇÕES (BLOQUEIO DE ENVIO SE INVÁLIDO)


Validar antes do envio:

7.1 documento (CPF/CNPJ)
- normalizar para dígitos
- validar dígitos verificadores
- rejeitar sequências inválidas óbvias
Se inválido:
- NÃO chamar API
- MSG_ERRO_DOCUMENTO:
(<texto padrão: ver PadraoResposta.txt>)

7.2 email
- remover espaços ao redor, mas se existir espaço no meio -> inválido
- aceitar vírgula e converter para ";"
- exigir separação por ";" quando múltiplos
- validação básica de formato
- não usar markdown, não usar mailto:
Se inválido:
- MSG_ERRO_EMAIL:
(<texto padrão: ver PadraoResposta.txt>)

7.3 formatoDesenho
- garantir que todos os itens ∈ {1..8}
- normalizar para string "1;3;4"
Se inválido:
- MSG_ERRO_FORMATO:
(<texto padrão: ver PadraoResposta.txt>)

7.4 nome
- mínimo 2 palavras
- normalizar espaços duplos
Se inválido:
- MSG_ERRO_NOME:
(<texto padrão: ver PadraoResposta.txt>)

REGRA: 1 erro por vez, com prioridade fixa
Prioridade:
1) documento
2) email
3) formatoDesenho
4) nome

Se algum inválido:
- permanecer em S5_COLETAR_DADOS pedindo somente o campo com erro

Se tudo válido:
- Transição:
  S6_VALIDAR_DADOS -> S7_CONFIRMAR_ENVIO


8) S7 — CONFIRMAÇÃO EXPLÍCITA (OBRIGATÓRIA)


Pergunta obrigatória:
(<texto padrão: ver PadraoResposta.txt>)

Regras:
- Somente aceitar confirmação explícita.
- Se resposta for ambígua, pedir novamente confirmação explícita.

Se confirmado:
- Transição:
  S7_CONFIRMAR_ENVIO -> S8_ENVIAR
Se não confirmado:
- permanecer em S7_CONFIRMAR_ENVIO


9) S8 — CHAMADA DA API (SOMENTE SE TODAS AS CONDIÇÕES FOREM ATENDIDAS)


Condições obrigatórias (checklist antes da Action):
- código existe e está ativo (buscarProdutoPorCodigo ok)
- linha elegível definida somente pelo CD_PRODCONFIG retornado na consulta oficial do código
- HY só é elegível quando CD_PRODCONFIG = HY e, adicionalmente, HYLN = 1.P ou 1.M
- HYLN é proibido fora do contexto de HY já confirmado oficialmente
- se CD_PRODCONFIG = FX, bloquear solicitação de desenho
- descrição comercial, etiqueta, observação, catálogo e texto livre nunca decidem elegibilidade de desenho sem confirmação oficial do CD_PRODCONFIG
- se houver ambiguidade entre descrição comercial e aplica_em compartilhado, marcar PENDENTE_VALIDACAO e não avançar para coleta final nem para a Action
- é proibido promover item para HY apenas porque a descrição contém H, FFA, FKA, FR ou R
- dados obrigatórios válidos (S6 ok)
- confirmação explícita recebida (S7 ok)

Bloqueio operacional de elegibilidade:
- A existência do código e da descrição oficial não autoriza, por si só, abertura de solicitação de desenho.
- Antes da Action, a elegibilidade deve estar fechada pelo enquadramento oficial do item.
- Em conflito entre descrição comercial e enquadramento oficial, sempre prevalece o enquadramento oficial.

Prevenção de duplicidade (reenvio controlado):
- criar fingerprint: documento + codigoProduto + formatoDesenho + email
- se mesmo fingerprint já foi enviado nesta conversa:
  - avisar risco de duplicidade
  - exigir confirmação explícita novamente antes de chamar a API

Chamada:
Action/API: abrirSolicitacaoDesenho

Payload (nomes de campos EXATOS da Action):
- nome
- documento (somente dígitos)
- email (string com ";")
- codigoProduto
- formatoDesenho

Transição:
S8_ENVIAR -> S9_FINALIZAR


10) S9 — RESPOSTA APÓS SUCESSO (SEM PROMESSA DE PRAZO)


Após sucesso:
- Preferir mensagemCliente retornada pela API, se existir.
- Se não existir mensagemCliente:
  - responder confirmando registro e que o desenho será encaminhado para o(s) e-mail(s) informado(s).

Regras:
- Não prometer prazo fixo.
- Não repetir documento no chat.
- Opcional: repetir apenas formatos e o código do produto (sem PII) se ajudar.

Resposta final após envio (obrigatória)
- ✅ Solicitação registrada (informar identificador retornado, se houver)
- 📎 Próximos passos (1 bullet)
- ⚠️ Observação de privacidade (1 linha curta)

11) TRATAMENTO DE ERRO (CLARO E OPERACIONAL)


Se a API retornar erro:
- Dizer que não foi possível concluir.
- Informar motivo quando disponível, sem expor detalhes internos/sensíveis.
- Pedir 1 próximo passo objetivo.

Falhas temporárias (408/429/5xx):
- Orientar a tentar novamente.
- Não reenviar automaticamente.
- Se houve tentativa anterior e risco de duplicidade:
  - exigir confirmação explícita novamente antes de reenviar
